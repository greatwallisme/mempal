# GUI / 桌面应用

## 本文件的特殊结构

Rust GUI 生态的概念锚点密度**分布不均**。通用 GUI 设计思想有充分的锚点覆盖（Raph Levien、egui、iced），但具体框架（特别是 Makepad）在模型训练数据中覆盖较薄。

因此本文件采用**混合策略**：
- 通用 GUI 设计哲学 → 概念锚点（释放层）
- 具体框架模式 → 源码样本（服从层）

当概念锚点的信息密度不够高时，源码样本就是外部实在——模型应该**服从示例代码的结构和风格**，而不是从稀疏的训练数据里猜。

## 概念锚点：通用 GUI 设计

### 状态管理的根本问题
参考 **Raph Levien**（druid → xilem）的系列博客：

Rust 做 GUI 的核心困难不是渲染，是**所有权和可变性与 GUI 状态模型的冲突**。
传统 GUI 框架（Qt、Cocoa）依赖对象图的双向引用——在 Rust 里这意味着
`Rc<RefCell<T>>` 满天飞，等于放弃了 Rust 的核心优势。

Raph Levien 在从 druid 到 xilem 的演化中探索的核心问题是：
如何在不放弃所有权安全性的前提下表达 GUI 的交互式状态？
他的结论趋向于**单向数据流 + 声明式视图函数**——
这和 React/SwiftUI 的方向殊途同归，但用 Rust 的类型系统做编译期保证。

### 三种范式的取舍

**Immediate Mode（即时模式）**—— 参考 **egui**（Emil Ernerfeldt）：
- 每帧完全重建 UI 描述，不维护持久的 widget 树
- 极简的 API：`ui.label("hello"); if ui.button("click").clicked() { ... }`
- 优点：和 Rust 所有权模型天然兼容，无双向引用
- 缺点：复杂布局困难、无法做细粒度重绘优化、辅助功能支持弱
- 适合：工具型 UI、调试面板、原型、内嵌编辑器

**Elm Architecture（TEA）**—— 参考 **iced**：
- Model-View-Message 三元组，单向数据流
- `update(msg) -> Command`，视图是状态的纯函数
- 优点：状态变化可预测、可调试
- 缺点：消息类型膨胀、深层嵌套组件间通讯不直观
- 适合：表单密集型应用、数据展示、需要强一致性的工具

**GPU-DSL 驱动（Makepad 模式）**：
- 所有渲染走 GPU shader，自定义 DSL 描述布局和样式
- Widget 树 + 事件系统 + live design（运行时修改 UI 不重编译）
- 优点：极致渲染性能、跨平台一致性、适合动画密集型应用
- 缺点：学习曲线陡、生态小、DSL 不是标准 Rust
- 适合：IDE、创意工具、音视频应用、需要自定义渲染的场景

选型判断：快速原型和工具型 → egui；数据驱动的桌面应用 → iced；
性能密集、动画密集、需要跨全平台的复杂应用 → Makepad。
